## 第1部 ・ ツールを再構築するプロンプト

### P1. 土台（単一ファイル・オフライン）
```
オフライン完結の単一HTMLツールを作って。要件：
- 1ファイルだけで動作。CDN・外部http参照ゼロ（OOXML名前空間URI・data URIのみ可）。API鍵は使わない。
- 左カラムに「①資料タイプ ②内容入力 ③配色 ④書き出し」の4カード、右にプレビュー。
- 入力・配色・設定は localStorage に自動保存。人物呼称はすべて「うさぎ先生」。
- 文体は WHY→WHAT→HOW、定義→たとえ→詳細→一言、ラーメン店の比喩でやさしく。
コードは読みやすく、後から型を足せる構造に。
```

### P2. 資料タイプ（12種）
```
資料タイプを切り替え式で12種そろえて。各タイプは {既定入力, 入力フォーム, プレビュー, .pptx/.docx 書き出し} を持つ：
研修オリエン／企画設計／時間割（学校風グリッド）／遠足しおり（=シラバス）／デザイン勉強会／
ワークショップ（問いを投げる）／口頭試験LPIC／筆記テスト／理解度クイズ／Word／全力サポート（テンプレ10）／汎用WWW。
スライド種別は cover/section/cards/rows/qa/grid を共通部品にして、各タイプはこの組合せで表現。
```

### P3. 受講生用／講師用トグル
```
筆記テストと理解度クイズに「受講生用／講師用」トグルを付けて。
- 受講生用：解答/答えを伏せる（プレビュー・PPT・Word・Markdownすべて）。
- 講師用：解答/答えを表示。
内部状態 audience を localStorage に保存。qa描画は ans が空なら解答ブロックを描かない実装に。
```

### P4. 自前OOXMLエンジン（依存なし）
```
依存ライブラリなしで .pptx と .docx を生成するエンジンを書いて。
- ZIP(stored)とCRC32を自前実装。PPTXは16:9（12192000×6858000 EMU）。
- 図形配置ヘルパ rect()/band() を用意。スライド種別ごとにshapesを構築。
- qa描画は汎用化：num が数値なら「第N問」、非数値はそのまま。hint行・任意ラベル・ans空なら解答非表示。
- DOCXは同じZIPエンジンを再利用し、Title/Heading1/Heading2/Normal スタイルで見出しはブランド色。
- すべてのテキストはXMLエスケープを通す。生成物は ECMA-376 準拠で PowerPoint/Word/LibreOffice で開けること。
```

### P5. デザイン・パレット（うさぎ画伯の秘密のパレット）
```
配色カードに「うさぎ画伯の秘密のパレット」を追加して：
- ネイティブカラーピッカー(<input type=color>) + カラーコード(#RRGGBB) + HSLスライダー(H/S/L)。三者は双方向同期。
- 色彩トーン10種（色みはそのままS/Lを変える）：ビビッド/ブライト/ストロング/ディープ/ソフト/ダル/ダーク/ペール/グレイッシュ/ライト。
- 役（背景/見出し/アクセント/文字）への「この色を入れる」ボタン。
- マイ絵具：選んだ色を登録（最大40・重複防止）し、クリックで役へ再利用、×で削除、localStorageに保存。
さらに色相環アシスト（類似±30/対照±150/補色+180）と、絵具比率70:25:5（合計100に正規化）も。
```

### P6. 仕上げ（使い方・画伯・名言）
```
- 「使い方」3〜4ステップの開閉セクションをヘッダー直下に。
- 右下に応援マスコット（画像はPNGをbase64で埋め込み）。クリックで海外の励まし名言を巡回表示、×で閉じ、一定間隔で自動めくり。
- ボタンは影＋縁取りで、押すとへこむ。入力欄は和紙風の質感（インラインSVGノイズ、通信なし）。
名言は短い定番のことばを、一般的に知られた出典名で（捏造しない）。
```

## 第2部 ・ 商用テストのプロンプト（査読論文・標準ベース）

> 各テストの前に必ず：「結果は実測のみ。未達は未達と書く。出典・数値を捏造しない」と指示する。

### T0. 定番（土台）
```
単一HTMLを商用品質で検査して（実測のみ）：
(1) <script>抽出→node --check  (2) jsdomで統合テスト  (3) 生成.pptx/.docxをpython-pptx/python-docxで開封
(4) LibreOfficeで実変換  (5) sha256で build=納品 同一性・外部CDN参照0・版数スタンプ実在
各pass/failを数値で。失敗は原因と影響まで。
```

### T1. 性質テスト/ファズ — Claessen & Hughes (2000), ICFP ’00
```
全生成経路に敵対的入力（< > & " ' </a:t> ]]> 絵文字 改行 空 超長文）を決定的シードで200回以上注入し、
性質を検査：例外なし／PK(ZIP)開始／zip整合／全XML well-formed／生の< > &がテキストに漏れない。
反例は最小化して提示。crash数/不正ZIP数/漏れ数を正直に。シードと注入文字も併記。
```

### T2. メタモルフィックテスト — Chen et al. (2018), ACM Comput. Surv. 51(1)
```
メタモルフィック関係(MR)を定義して検査：
MR1 加法性（設問+1→qa+1）／MR2 受講生用⊆講師用（答え秘匿の単調性）／
MR3 配色不変（色を本文に出さない型）／MR3' 色を教材に出す型は色変更で本文が変わるのが正。
MRが破れたら「バグ」か「MR記述ミス」かを切り分けて明示。
```

### T3. 決定性／再現ビルド
```
同一モデルから .pptx/.docx を2回生成しバイト一致を確認。タイムスタンプ・乱数・反復順などの非決定要因を点検。
```

### T4. WCAG 2.1 コントラスト監査 — W3C WCAG 2.1 ／ 国内：伊藤啓(2012)・日本色彩学会誌42(5)2018
```
全パレット（タイプ既定＋プリセット）のhexを抽出し、相対輝度式でコントラスト比を計算：
本文 text/bg・text/白 → AA 4.5:1（AAAは7:1）／見出し main/bg・main/白 → AA(大) 3:1。
数値と合否を表で。未達は「どのパレットのどのペアが何:1」で列挙。AAをAAAと水増ししない。
色覚多様性の観点（色だけでなく明度差で区別できているか）も一言。
```

### T5. OOXML 整合 — ECMA-376 / ISO-IEC 29500
```
全 .pptx/.docx で：[Content_Types].xml が全パートを網羅／全 .rels の内部Targetが実在／dangling 0 を検査。
不整合は「どのファイルのどのパート/Target」まで具体に。
```

### T6. HTML 適合 — WHATWG / html5lib 1.1
```
配布HTMLを html5lib で解析しパースエラーを分類集計。
重大（duplicate-attribute / eof-in-tag / 不正タグ名等）が0であることを確認。軽微は件数と内訳を開示。
```
