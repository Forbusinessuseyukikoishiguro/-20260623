## 1. 要件定義

### 1.1 背景・目的（WHY）
研修講師が、講義・演習・試験・配布物などの教材を「見やすく・速く・配布可能な形式で」作るためのツール。受講生を一律に「うさぎ先生」とし、親しみやすい比喩（うさうさラーメン店）と WHY→WHAT→HOW 構成で、教材作成のハードルを下げる。

### 1.2 機能要件（WHAT）
1. **12種の資料タイプ**を選び、内容を入力すると、スライド/文書のプレビューが即時生成される。
   - 研修オリエン／企画設計／時間割／遠足しおり（＝シラバス）／デザイン勉強会／ワークショップ／口頭試験LPIC／筆記テスト／理解度クイズ／Word／全力サポート（テンプレ10）／汎用WWW。
2. **配色をえらぶ**：プリセット、役（背景/見出し/アクセント/文字）への割り当て、30スウォッチ、色相環アシスト、絵具比率（70:25:5）、**うさぎ画伯の秘密のパレット**（カラーピッカー＋カラーコード＋HSL＋色彩トーン＋マイ絵具）。
3. **書き出し**：📊PowerPoint(.pptx)／📄Word(.docx)／📥HTML／📋Markdown／🖨印刷／📝空ひな型DL／🐞バグ相談プロンプト。
4. **受講生用／講師用トグル**（筆記テスト・理解度クイズ）：受講生用は解答/答えを伏せ、講師用は表示。プレビュー・PPT・Word・Markdownすべてに反映。
5. **使い方セクション**（3〜4ステップの案内）と、右下の**うさうさ画伯**（海外の励まし名言を巡回表示する応援マスコット）。
6. **自動保存**：入力・配色・トグル・マイ絵具を localStorage に保存。

### 1.3 非機能要件
- **オフライン／単一ファイル**：1つの .html だけで動作。CDN・外部http参照ゼロ（OOXML名前空間URI・data URI のみ可）。
- **API鍵はメモリのみ**：鍵をファイルやlocalStorageに保存しない（本ツールは外部APIを使わないため鍵自体不要）。
- **嘘をつかない原則**：存在しない出典・結果を作らない。生成物・検査結果は実測のみ。
- **アクセシビリティ**：本文/見出しの配色は WCAG 2.1 コントラストを満たす（§7参照）。色だけに頼らず明度差で区別。
- **互換性**：生成 .pptx/.docx は ECMA-376 準拠で、PowerPoint/Word/LibreOffice で開ける。

## 2. 技術仕様（アーキテクチャ）

### 2.1 全体構成
- 単一HTML。`<style>`（静的CSS）＋ 2つの実行 `<script>` ブロック。
  - 第1 `<script>`：アプリ本体（状態・描画・各タイプ生成・配色・画伯・名言）。
  - 第2 `<script>`：自前OOXML/ZIPエンジン（`window.UsausaPPTX`）。
- 画伯画像はPNGを base64 data URI で埋め込み（`PAINTER_IMG`）。ロゴうさぎも base64（`MASCOT`）。

### 2.2 ビルド系（ソース→納品）
編集対象はソース群。納品物はプレースホルダ注入で生成する。
- `tool.src.html`：本体（プレースホルダ `__MASCOT__` / `/* __PPTXGEN__ */` / `__PAINTER__` を含む）。
- `pptxgen.js`：OOXMLエンジン（`/* __PPTXGEN__ */` に注入）。
- `mascot.txt`：ロゴうさぎ base64（`__MASCOT__` に注入）。
- `painter_b64_keep.txt`：画伯 data URI（`__PAINTER__` に注入）。
- ビルド：上記を置換 → `うさうさ_研修資料ジェネレーター.html` を出力。注入後はプレースホルダが残っていないことを検査。

### 2.3 データモデル（主な構造）
- `state`：`{type, fields, pals, ratio, audience, pick, myPaints, idx, role}`
- `DEF`：タイプ別の既定入力（タイトル・本文など）。
- `PAL_DEF`：タイプ別の既定配色 `{bg, main, accent, text}`（hex）。
- `PRESETS`：プリセット5種 `{n:"名前", p:{bg,main,accent,text}}`（和ざくら/藍そう/翠みどり/古今/墨）。
- `TONES`：色彩トーン10種 `{n, s, l}`（彩度・明度の目標）。
- `QUOTES`：海外の励まし名言10種 `{ja, en, by}`。
- `TEMPLATES`：全力サポートのテンプレ10種 `{name, fmt, type, f, note}`。

### 2.4 永続化
- localStorage キー：`usausa_shiryo_v2`
- 保存内容：`{type, fields, pals, ratio, audience, myPaints, pick}`。起動時に読み戻す（無ければ既定）。

- `buildModel()`：書き出し用モデル `{title, version, pal, slides[]}` を返す。
- `deckCSS()`：プレビュー＆HTML書き出し両用のスライドCSS文字列。
- `anyToWordModel()`：任意タイプ→ .docx 用ブロックモデルへ。
- `wordToBlocks(body)`：Word本文記法（# 見出し等）→ブロック。
- 各タイプの specs：`timetableSpecs / shioriSpecs / workshopSpecs / examSpecs / quizSpecs / planSpecs / designSpecs …`。
- 秘密のパレット：`initSecret / setPick / pickFromSliders / toneApply / renderTones / renderMyPaints`。

## 4. 実装仕様

### 4.1 OOXMLエンジン（`window.UsausaPPTX`）
- 公開：`buildPPTX(model)`, `buildDOCX(model)`, `zip(files)`, `crc32(bytes)`。
- ZIP：stored（無圧縮）でローカルファイルヘッダ＋セントラルディレクトリを自前生成。CRC32は自前計算。
- PPTX：16:9（12192000 × 6858000 EMU）。`rect()/band()` ヘルパで図形を配置。スライド種別ごとにshapesを構築。
- `qa` 描画は汎用化：`num` が数値なら「第N問」、非数値はそのまま接頭。`hint` 行、ラベルは `lbl`（既定「模範解答」）、`ans` が空なら解答ブロックを描かない（受講生用で解答を伏せる仕組み）。
- DOCX：同じZIPエンジンを再利用。Yu Gothic、Title/Heading1/Heading2/Normal スタイル、見出しはブランド色。

### 4.2 エスケープ／配色変数
- すべてのテキストは XML/HTML エスケープを通す（`< > & " '`）。
- 配色は CSS変数（`--dk-bg/--dk-main/--dk-accent/--dk-text`）に展開し、プレビューと書き出しで共有。

### 4.3 受講生用／講師用
- `state.audience ∈ {受講生用, 講師用}`。`examSpecs/quizSpecs` は `show = audience==="講師用"` のときのみ `ans` を設定。`anyToWordModel`・`mdCopy` も同様に分岐。

## 5. デザイン・パレット仕様（うさぎ画伯の秘密のパレット）

### 5.1 役と基本操作
- 役：背景(bg)／見出し(main)／アクセント(accent)／文字(text)。役を選び、色をタップ→その役に反映。
- スウォッチ：30色。プリセット：5種（§2.3）。

### 5.2 色相環アシスト
- 見出し色をシードに、HSLの色相回転で配色候補を提示。
  - 類似色 = ±30°、対照色 = ±150°、補色 = +180°。
- ボタン：「補色をアクセントに」「類似色を背景に」。

### 5.3 絵具の比率（70:25:5）
- デザイン勉強会で、ベース:メイン:アクセント の比率を増減（合計は100に自動正規化）。配色スライドに反映。

### 5.4 秘密のパレット（絵具タブ）
- **ネイティブカラーピッカー**（`<input type=color>`）で世界中の色を直観選択。
- **カラーコード**入力（`#RRGGBB`）。ピッカー・スライダーと双方向同期。
- **HSLスライダー**：色相H(0–360)／彩度S(0–100)／明度L(0–100)。
- **色彩トーン10種**（色みはそのまま、S/Lを変える）：

| トーン | S | L | | トーン | S | L |
|---|---|---|---|---|---|---|
| ビビッド | 95 | 50 | | ダーク | 55 | 26 |
| ブライト | 78 | 62 | | ペール | 28 | 87 |
| ストロング | 85 | 46 | | グレイッシュ | 14 | 52 |
| ディープ | 82 | 32 | | ライト | 55 | 73 |
| ソフト | 42 | 62 | | ダル | 32 | 52 |

- **「この色を“役”に入れる」**：現在の選択色を選択中の役へ適用。
- **マイ絵具（カスタマイズ＝既定登録）**：気に入った色を登録（最大40・先頭追加・重複防止）。クリックで役へ再利用、×で削除。**localStorage に保存**し次回も残る。

### 5.5 色彩理論の根拠
- 色彩トーンは PCCS 的なトーン概念（明度・彩度の組）に基づく実務的な近似。
- 配色のアクセシビリティは WCAG 2.1（§7）と、色覚多様性に関する国内研究（§9 参考文献）に依拠し、**色だけでなく明度差**で区別できることを重視。

## 7. 新テスト手法（査読論文・国際標準ベース）

> 「これまでやっていない」観点を、実在する査読論文・標準に基づき設計。結果は実測のみ。

### 7.1 性質テスト/ファズ（Property-based / Fuzz）
- 根拠：**Claessen & Hughes (2000), ICFP ’00**。
- 方法：敵対的入力（XMLメタ文字・絵文字・空・超長文）を決定的シードで220回注入。
- 性質：例外なし／PK(ZIP)で始まる／zip整合／全XML well-formed／**生の `< > &` がテキストに漏れない**。
- 実測：crash 0／不正ZIP 0／漏れ 0。

### 7.2 メタモルフィックテスト（Metamorphic Testing）
- 根拠：**Chen et al. (2018), ACM Computing Surveys 51(1)**。
- 関係：MR1 加法性（設問+1→qa+1）／MR2 受講生用⊆講師用（答え秘匿の単調性）／MR3 配色不変（色を本文に出さない型）／MR3′ デザインは色＝内容なので色変更で本文が変わる（仕様）。
- 実測：全MR成立（MR3はデザインで一度「破れ」を検出→“関係の記述ミス”と切り分けて修正、の経緯を含む）。

### 7.3 決定性／再現ビルド
- 同一入力から .pptx/.docx を2回生成し**バイト一致**。非決定要因なし。

### 7.4 WCAG 2.1 コントラスト監査
- 根拠：**W3C WCAG 2.1**（SC 1.4.3：通常4.5:1／大3:1、SC 1.4.6：7:1／4.5:1）。国内：**伊藤啓 (2012)**、**日本色彩学会誌 42(5) 2018**。
- 方法：全17パレット（タイプ既定12＋プリセット5）の hex を抽出し相対輝度式でコントラスト比を算出。
- 実測：**本文 text は17/17でAA(4.5)かつAAA(7.0)達成**、見出し main は17/17でAA(大3.0)達成。最も細いのは遠足しおりの見出し（main/bg=3.15、main/白=3.37）。

### 7.5 OOXML 整合（ECMA-376 / ISO/IEC 29500）
- `[Content_Types].xml` が全パートを網羅／全 `.rels` の内部 Target が実在／dangling 0。実測：不整合0。

### 7.6 HTML 適合（WHATWG / html5lib 1.1）
- 配布HTMLを html5lib で解析しパースエラーを分類集計。実測：パースエラー総数0。

### 7.7 新テストの実測スナップショット（2026-06-23）
- Node（A/B/C）：21 pass / 0 fail
- Python（D/E/F）：4 pass / 0 fail
- 合計：**25項目 pass / 0 fail**（定番89項目とは別枠）。

## 8. 再現用仕様書（ゼロから再構築する手順）

### 8.1 ファイル構成
```
build/
├ tool.src.html            # 本体（プレースホルダ入り）
├ pptxgen.js               # OOXMLエンジン（window.UsausaPPTX）
├ mascot.txt               # ロゴうさぎ base64
├ painter_b64_keep.txt     # 画伯 data URI
├ test_jsdom.js            # 定番・統合テスト（89項目）
├ ctest_node.js            # 新観点：性質/メタモルフィック/決定性
└ ctest_py.py              # 新観点：WCAG/OOXML整合/HTML適合
```

### 8.2 ビルド手順
1. `tool.src.html` の `__MASCOT__` / `/* __PPTXGEN__ */` / `__PAINTER__` を各ファイル内容で置換。
2. プレースホルダ残存がないことを確認し、`うさうさ_研修資料ジェネレーター.html` を出力。
3. 検証：`node --check`（2ブロック）→ `node test_jsdom.js`（89/0）→ `node ctest_node.js`（21/0）→ クリーン生成物を出し `python3 ctest_py.py`（4/0）。
4. 生成 .pptx/.docx を python-pptx/python-docx で開封、LibreOffice で実変換。
5. sha256 で build＝納品の同一性、刻印4箇所、外部CDN参照0 を確認。

### 8.3 復元の頑健性
- 万一 `pptxgen.js` を失っても、納品HTMLの**第2 `<script>` ブロック**＝エンジン本体なので、そこから復元できる（復元ソースで再ビルドすると sha256 が一致することを確認済み）。

### 8.4 リリース手順
- 機能追加→テスト→版数スタンプ更新（4箇所）→再ビルド→最終テスト→納品＋sha256記録。
